# STRICT TDD RULES

For **every single feature**, follow this exact workflow.

## Step 1 — RED

Write the failing test first.

The implementation must not exist yet.

Run the tests.

Verify they fail.

Do not continue until the failure is confirmed.

---

## Step 2 — GREEN

Write the smallest amount of production code necessary.

Do not add extra features.

Only satisfy the failing test.

Run the tests.

Verify they pass.

---

## Step 3 — REFACTOR

Improve the implementation while keeping all tests green.

Remove duplication.

Improve naming.

Improve readability.

Keep SOLID principles.

Run the tests again.

---

Repeat this cycle for **every authentication feature**.

Never implement multiple features before writing tests.

# Features To Implement

Implement only the following.

## User Registration

Implement

POST /api/auth/register

Requirements

- Register a new user
- Validate request body with Zod
- Hash password using bcrypt
- Store user in PostgreSQL using Prisma
- Never store plain text passwords
- Prevent duplicate email registration
- Return appropriate HTTP status codes
- Return safe user information only
- Never return password hash

---

## User Login

Implement

POST /api/auth/login

Requirements

- Validate request body
- Find user by email
- Compare password using bcrypt
- Reject invalid credentials
- Generate JWT access token
- Generate refresh token
- Return tokens
- Return authenticated user information
- Never expose password hash

---

## Password Hashing

Use bcrypt.

Requirements

- Secure password hashing
- Secure password comparison
- Configurable salt rounds through environment variables

---

## JWT Authentication

Requirements

- Generate JWT access tokens
- Configurable expiration
- Secret loaded from environment variables
- Verify JWT
- Reject invalid tokens
- Reject expired tokens
- Reject malformed tokens

- Reject expired refresh tokens
